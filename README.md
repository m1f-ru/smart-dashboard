# Дашборд СМАРТ — клиентская (статическая) версия для Bitrix24

Это самодостаточная версия дашборда СМАРТ, рассчитанная на встраивание в **Bitrix24 (облако)**,
где нельзя выполнять серверный код. Весь UI — статика, данные собираются заранее в GitHub Actions.

Как это работает:

- **GitHub Action** (`.github/workflows/refresh-data.yml`) раз в 6 часов запускает Node-экстрактор
  `scripts/build-data.mjs`, который ходит в 1С:Fresh (УНФ + Бухгалтерия) по OData и собирает
  компактный снимок `data.json` (агрегированные числа, ~5–15 КБ), после чего коммитит его
  в репозиторий.
- **GitHub Pages** публикует **корень репозитория** (Source: `/ (root)`) и отдаёт `data.json` с заголовком
  `Access-Control-Allow-Origin: *`, поэтому CORS не мешает читать его из браузера.
- **Клиент** (`index.html` + `css/` + `js/`) подтягивает `data.json` по `fetch`
  и рисует весь UI на JavaScript. Переключение года и режим «Все года» считаются на клиенте.

Данные обновляются автоматически раз в 6 часов. Пароли 1С нигде, кроме GitHub Secrets, не хранятся
и в `data.json` не попадают.

## Структура репозитория

Содержимое каталога `web/` — это **корень** публикуемого репозитория.

```
.                          ← корень репозитория = публикуется через GitHub Pages (Source: / root)
├── index.html             ← страница дашборда (встраивается в Bitrix24 через iframe)
├── css/                   ← стили
├── js/                    ← клиентская логика (рендер, графики, переключение года)
├── assets/                ← иконка, фолбэк-копия Chart.js, favicon
├── data.json              ← снимок данных из 1С (генерируется автоматически)
├── robots.txt
├── scripts/               ← Node-экстрактор (в браузер не попадает)
│   ├── build-data.mjs     ← OData → data.json
│   └── lib/               ← модули порта (odata, unf, buh, …)
├── package.json           ← метаданные проекта (зависимостей нет, zero-dep)
├── .github/workflows/
│   └── refresh-data.yml    ← обновление данных по расписанию и по кнопке
├── README.md
└── .gitignore
```

> Проект **zero-dep**: внешних npm-пакетов нет, `npm install` не требуется. Достаточно Node 20+.

## Как развернуть

### 1. Создать репозиторий и запушить содержимое `web/` как корень

Содержимое каталога `web/` — это **корень** будущего репозитория. Пушим именно содержимое `web/`
(то есть `index.html`, `css/`, `js/`, `assets/`, `data.json`, `scripts/`, `package.json`, `.github/`, `README.md`, `.gitignore`), а не саму папку `web/`.

```bash
# находясь внутри web/
git init
git add .
git commit -m "init: статическая версия дашборда СМАРТ"
git branch -M main
git remote add origin https://github.com/USER/REPO.git
git push -u origin main
```

### 2. Завести секреты и переменные (Settings → Secrets and variables → Actions)

**Secrets** (вкладка *Secrets*) — креды **урезанного пользователя 1С** (только чтение + публикация
в OData) и GUID организации СМАРТ в каждой базе. Нужны все 8:

| Секрет | Что это |
| --- | --- |
| `ODATA_BASE_UNF` | Базовый URL OData-сервиса базы **УНФ** (1С:Управление нашей фирмой) |
| `ODATA_USER_UNF` | Логин урезанного пользователя 1С в базе УНФ |
| `ODATA_PASS_UNF` | Пароль этого пользователя в базе УНФ |
| `ORG_FILTER_UNF` | GUID (Ref_Key) организации СМАРТ в базе УНФ |
| `ODATA_BASE_BUH` | Базовый URL OData-сервиса базы **Бухгалтерии** |
| `ODATA_USER_BUH` | Логин урезанного пользователя 1С в базе Бухгалтерии |
| `ODATA_PASS_BUH` | Пароль этого пользователя в базе Бухгалтерии |
| `ORG_FILTER_BUH` | GUID (Ref_Key) организации СМАРТ в базе Бухгалтерии |

**Variables** (вкладка *Variables*) — необязательные несекретные параметры:

| Переменная | Назначение | Пример |
| --- | --- | --- |
| `REPORT_YEARS` | Список лет для отчёта (через запятую) | `2026,2025,2024` |
| `REPORT_YEAR` | Год по умолчанию | `2025` |

Если переменные не заданы, берутся три последних года (текущий и два предыдущих), а год по умолчанию — максимальный.

### 3. Включить GitHub Pages

Settings → **Pages** → Source: **Deploy from a branch** → ветка **main**, папка **`/ (root)`**. Сохранить.

### 4. Запустить workflow вручную и проверить данные

Actions → **refresh-data** → **Run workflow**. Дождаться зелёного прогона и убедиться, что в репозитории
появился/обновился коммит `chore: refresh dashboard data [skip ci]` с изменённым `data.json`.

### 5. Проверить дашборд по Pages-URL

Открыть `https://USER.github.io/REPO/` — должна отрендериться автономная страница (`index.html`)
со всеми блоками. Это контрольная проверка перед встраиванием в Bitrix24.

### 6. Встроить в Bitrix24 через `<iframe>`

Вся страница уже опубликована на Pages (`index.html`). В Bitrix24 вставляется только маленький
блок с `<iframe>` на ваш Pages-адрес. Страница сама сообщает свою высоту (`postMessage`), а
крошечный скрипт рядом подгоняет высоту iframe — без внутренней полосы прокрутки, высота
меняется при смене года/фильтров/«Скрыть данные». Никаких URL внутри страницы менять не нужно —
`index.html` уже берёт `./data.json`.

Вставьте в Bitrix24 (замените `USER`/`REPO` на свои):

```html
<iframe id="smart-dash-frame"
        src="https://USER.github.io/REPO/"
        style="width:100%;border:0;display:block;min-height:600px"
        scrolling="no" loading="lazy"
        title="Аналитика СМАРТ"></iframe>
<script>
(function () {
  var ORIGIN = "https://USER.github.io"; // ← ваш origin Pages (для безопасности)
  window.addEventListener("message", function (e) {
    if (e.origin !== ORIGIN) { return; }              // принимаем только от своего Pages
    if (!e.data || e.data.type !== "sd-height") { return; }
    var f = document.getElementById("smart-dash-frame");
    if (f && e.data.height) { f.style.height = e.data.height + "px"; }
  });
})();
</script>
```

> Примечание: GitHub Pages (`*.github.io`) не запрещает встраивание в iframe (нет
> `X-Frame-Options`/ограничивающего `frame-ancestors`), поэтому страница отображается внутри
> Bitrix24. Если в вашем HTML-виджете Bitrix24 вырезает `<script>`, оставьте только сам
> `<iframe>` и задайте фиксированную высоту (напр. `height:4200px`) — тогда авто-подгон высоты
> не работает, но дашборд отображается.

## Минимальные права пользователя 1С

Пользователь 1С должен быть **урезанным**: только **чтение** перечисленных объектов + **публикация
в стандартном интерфейсе OData** + доступ к организации **СМАРТ (ИНН 9727079722)**.

**УНФ (Управление нашей фирмой):**

- `Catalog_Контрагенты`
- `Catalog_СостоянияЗаказовПокупателей`
- `Document_ЗаказПокупателя`
- `Document_СчетНаОплату`
- `Document_РасходнаяНакладная`
- `AccumulationRegister_ОплатаСчетовИЗаказов` (Turnovers)

**Бухгалтерия:**

- `AccountingRegister_Хозрасчетный` (Turnovers + Balance)
- `ChartOfAccounts_Хозрасчетный`
- `Document_РегламентированныйОтчет` (нужен для сверки с ОФР)

Все объекты — **на чтение**, опубликованы в стандартном OData-интерфейсе, доступ ограничен организацией СМАРТ.

## Безопасность

- **Пароли 1С — только в GitHub Secrets.** В `data.json` их нет, в логах workflow они
  не печатаются. Не храните креды в репозитории (локальный конфиг — `config.local.php`/`.env` — в `.gitignore`).
- **`data.json` публично содержит реальные финансовые данные СМАРТ.** GitHub Pages — публичный ресурс:
  любой, у кого есть ссылка, увидит цифры. Давайте ссылку осознанно. Если публичность недопустима —
  **держите репозиторий приватным** (тогда Pages будет доступен только при соответствующем тарифе/настройке).
- **Хост-страница Bitrix24 должна быть за авторизацией** — встраивайте блок только на страницы,
  доступные авторизованным сотрудникам.

## Как обновить данные вручную

- **Через GitHub:** Actions → **refresh-data** → **Run workflow**.
- **Локально:** задайте переменные окружения OData (`ODATA_*_UNF`/`ODATA_*_BUH`, `ORG_FILTER_*`,
  опционально `REPORT_YEARS`/`REPORT_YEAR`) и выполните:

  ```bash
  node scripts/build-data.mjs
  ```

  Скрипт перезапишет `data.json`. Зависимостей нет — достаточно Node 20+, `npm install` не нужен.
